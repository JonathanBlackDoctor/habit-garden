# 기도제목 관리 시스템 — 설계 문서

> 습관 정원(Habit Garden) PWA에 추가하는 "기도제목 관리" 기능의 전체 설계.
> 기존 스택(React + TS + Firebase + Gemini 2.0 Flash)을 그대로 활용한다.
> 작성일: 2026-05-21

---

## 1. 목표

기도 요청을 **무노력에 가깝게 저장**하고, **매일 기도 여부를 체크**하며, **무더기·비정형으로 들어오는 여러 명의 기도제목을 AI가 자동 정리**하는 개인 전용 시스템.

핵심 3대 기능:

1. **빠른 저장** — 한 줄만 적어도 바로 저장. 분류·우선순위는 나중에 보강하거나 AI가 추정.
2. **기도 여부 체크** — 매일 "오늘의 기도 목록"을 체크하며 스트릭을 쌓고, 응답되면 기록으로 남긴다.
3. **무더기 자동 정리** — 통일되지 않은 텍스트 덩어리를 붙여넣으면 AI가 사람별·항목별로 쪼개 양식에 맞춘다.

---

## 2. 확정된 설계 결정

| # | 항목 | 결정 |
|---|------|------|
| 1 | 기도 여부 체크 방식 | **매일 체크(스트릭) + 응답 기록** |
| 2 | 정리 기준 | **사람(대상)별 + 분류 + 받은 날짜** (3중 메타데이터) |
| 3 | 무더기 입력 방식 | **텍스트 붙여넣기 → AI 자동 정리** |
| 4 | 빠른 저장 위치 | **하단 탭에 전용 "기도" 탭 신설** |
| 5 | 오늘의 목록 구성 | **자동 로테이션 + 고정**, 오래되거나 중요도 낮은 것은 점점 망각 |
| 6 | AI 정리 후 처리 | **검토 후 저장** (미리보기 → 수정/삭제 → 일괄 저장) |
| 7 | 응답 처리 | **응답 보관함으로 이동** (활성 목록에서 빠짐, 간증 보관) |
| 8 | 게임화 연동 | **포인트·스트릭 연동** (기존 경제 시스템과 통합) |
| 9 | 망각 처리 | 노출 빈도가 점점 줄다가 **"잠든 기도" 보관함**으로 이동 (부활 가능) |

## 3. 데이터 모델 (Firestore)

기존 `users/{uid}/...` 단일 사용자 구조를 따른다. (uid 고정, `asia-northeast3`)

### 3.1 기도 대상자 — `users/{uid}/people/{personId}`

여러 명의 기도제목을 "사람별"로 묶기 위한 가벼운 인물 레코드.

```ts
export interface PrayerPersonDoc {
  id: string;
  name: string;                 // 표시 이름
  aliases?: string[];           // 다른 표기(별명 등) — AI 중복 매칭용
  relation: PrayerCategory;     // 기본 분류 (아래 category와 동일 enum)
  note?: string;                // 메모(관계, 배경)
  activeCount: number;          // 활성 기도제목 수 (denormalized)
  answeredCount: number;        // 응답된 기도제목 수
  createdAt: Timestamp;
  updatedAt: Timestamp;
}
```

> "자기 자신/일반"처럼 사람이 특정되지 않는 기도제목도 있으므로 `personId`는 선택값이다.
> 사람 없이 저장된 항목은 `personName`(비정규화 필드)만 갖는다.

### 3.2 기도제목 — `users/{uid}/prayers/{prayerId}`

기존 `PrayerDoc`을 확장. (기존 필드: category, title, body, rotationDays, priority, active)

```ts
export type PrayerCategory = 'self' | 'family' | 'church' | 'ministry' | 'friend' | 'other';
export type PrayerStatus   = 'active' | 'answered' | 'dormant';   // 활성 / 응답됨 / 잠든

export interface PrayerDoc {
  id: string;

  // ── 정리 기준 (결정 #2) ─────────────────────
  personId?: string;            // 대상자 링크 (없을 수 있음)
  personName: string;           // 표시·검색용 (비정규화)
  category: PrayerCategory;     // 분류
  receivedAt: Timestamp;        // 받은 날짜

  // ── 내용 ───────────────────────────────────
  title: string;                // 한 줄 요약 (목록 표시)
  body?: string;                // 상세 / 원문 보존
  tags?: string[];              // 자유 태그(선택)

  // ── 우선순위·로테이션 (결정 #5) ──────────────
  priority: 'high' | 'mid' | 'low';
  pinned: boolean;              // 고정 = 매일 노출, 망각 안 됨
  rotationDays?: number;        // 희망 주기(일). 미지정 시 priority 기본값 사용

  // ── 상태·추적 (결정 #1, #7, #9) ──────────────
  status: PrayerStatus;
  lastPrayedAt?: Timestamp;     // 마지막으로 기도한 시각
  prayCount: number;            // 누적 기도 횟수
  streak: number;               // 이 제목 연속 기도(선택적 표시)
  answeredAt?: Timestamp;
  answerNote?: string;          // 응답 간증
  dormantSince?: Timestamp;     // 잠든 시각

  // ── 출처·메타 ───────────────────────────────
  source: 'quick' | 'manual' | 'bulk_ai';
  createdAt: Timestamp;
  updatedAt: Timestamp;
}
```

> 기존 `active: boolean`은 `status`로 대체한다(active = status==='active'). 마이그레이션 시 변환.

### 3.3 일일 기도 체크 — `users/{uid}/days/{YYYY-MM-DD}/prayerChecks/{prayerId}`

기존 `habitChecks` 패턴을 그대로 차용 → 게임화 트리거(`onWrite`)와 히스토리에 자연스럽게 연결.

```ts
export interface PrayerCheckDoc {
  prayerId: string;
  prayedAt: Timestamp;
}
```

### 3.4 일일 기도 계획 — `users/{uid}/days/{YYYY-MM-DD}` 문서에 필드 추가

`dailyReset`(04:00 KST)이 매일 "오늘의 목록"을 미리 계산해 저장.

```ts
// DayDoc 에 추가
prayerPlan?: {
  pinnedIds: string[];      // 고정 — 항상 노출
  rotationIds: string[];    // 오늘 추천된 로테이션 목록 (상한 N개)
  generatedAt: Timestamp;
};
```

### 3.5 진척 문서 확장 — `users/{uid}/progress`

```ts
// ProgressDoc 에 추가
prayerStreak?: number;        // 기도 전역 연속일
prayerBestStreak?: number;
lastPrayerDate?: string;      // 'YYYY-MM-DD'
totalPrayersAnswered?: number;
```

## 4. 화면 구성

하단 탭바에 **"기도" 탭**(아이콘: `HandHeart` 또는 `Heart`, lucide-react)을 신설한다.

> ⚠️ **UI 결정 필요(경미):** 현재 탭이 5개(오늘·습관·정원·진척·더보기)다. 6번째 탭을 추가하면 모바일에서 다소 빽빽해진다.
> 권장안: 6탭 유지하되 라벨을 짧게. 너무 좁으면 "진척"을 더보기로 내리는 대안도 가능. → 시작 시 확인.

기도 탭 내부는 상단 세그먼트 컨트롤로 4개 뷰 + 2개 진입점.

```
┌─────────────────────────────────────┐
│  [+ 빠른 추가 한 줄 입력칸........]  │   ← 항상 상단, 무노력 저장
├─────────────────────────────────────┤
│  [오늘] [전체] [응답] [잠든]          │   ← 세그먼트
│                          [📋 무더기]  │   ← 무더기 정리 진입
└─────────────────────────────────────┘
```

### 4.1 오늘 (기본 화면)

- **진행 표시**: "오늘 3 / 9 기도" 프로그레스 바.
- **📌 고정** 섹션: pinned 기도제목 (항상 노출).
- **오늘의 로테이션** 섹션: 알고리즘이 고른 오늘 기도할 목록.
- 각 항목: `[ ○ ]  대상자 · 제목` — 동그라미 탭 → 기도 완료(체크), 포인트 적립, 살짝 애니메이션.
- 길게 누르면 상세(상세내용, 우선순위, 고정, 응답 처리).
- 모두 체크하면 "오늘 기도 완료" → 스트릭 +1, 보너스 포인트.

### 4.2 빠른 추가 (결정 #4 — 무노력 저장)

- 상단 한 줄 입력칸 + `+` 버튼. **제목만으로 즉시 저장.**
- 기본값: `category = 직전 사용값 or 'other'`, `priority = 'mid'`, `status = 'active'`, `receivedAt = now`, `source = 'quick'`.
- 저장 직후 토스트에 "분류/대상 지정" 빠른 보강 링크(선택). 무시해도 저장은 끝난 상태.

### 4.3 무더기 정리 (결정 #3, #6 — AI 자동 정리 + 검토)

1. "📋 무더기" → 큰 텍스트 영역에 카톡/메모 덩어리 붙여넣기.
2. "정리하기" → Cloud Function `parsePrayerBulk` 호출(Gemini).
3. **검토 화면**: 사람별·항목별로 쪼개진 카드 목록이 미리보기로 표시.
   - 각 카드: 대상자(기존 인물과 자동 매칭 표시) · 분류 · 우선순위 · 제목 · 상세 — 모두 인라인 수정 가능.
   - 포함/제외 체크박스. 오분류 의심 항목은 노란 배지로 표시.
4. "N개 저장" → 한 번에 기록. 신규 대상자는 `people`에 자동 생성.

### 4.4 전체

- 필터: 사람별 · 분류별 · 받은 날짜순 · 우선순위 (결정 #2 전부 지원).
- 검색(제목/대상/태그). 사람 칩 탭 → 그 사람의 모든 기도제목.

### 4.5 응답 (응답 보관함 — 결정 #7)

- 응답된 기도제목 목록 + 간증(answerNote) + 응답 날짜.
- 활성 목록·로테이션에서는 빠져 있고, 여기서만 돌아본다.

### 4.6 잠든 (잠든 기도 보관함 — 결정 #9)

- 망각되어 내려간 항목. "깨우기" 버튼 → status를 active로 되돌리고 lastPrayedAt 리셋.
- 검색 가능. 영구 삭제도 여기서.

## 5. 로테이션 & 망각 알고리즘 (결정 #5, #9)

매일 `dailyReset`에서 계산하여 `prayerPlan`에 저장. (클라이언트 동일 로직 fallback 가능)

### 5.1 기본 주기

`rotationDays`가 지정돼 있으면 그 값을 우선. 없으면 우선순위 기본값:

| priority | 기본 주기(baseInterval) | 잠듦 임계(dormantThreshold) |
|----------|------------------------|----------------------------|
| high | 2일 | 120일 |
| mid  | 5일 | 75일 |
| low  | 10일 | 45일 |

### 5.2 망각(점진적 backoff)

오래된 요청일수록 노출 주기가 점점 늘어난다 → "점점 망각".

```
ageWeeks         = floor((today - receivedAt) / 7)
backoffFactor    = 1 + floor(ageWeeks / 4) * 0.5        // 4주마다 +50%
effectiveInterval = baseInterval * backoffFactor
daysSince        = today - (lastPrayedAt ?? receivedAt)
dueScore         = daysSince / effectiveInterval         // ≥ 1 이면 "오늘 대상"
```

### 5.3 오늘의 목록 선정

```
1) pinned = true        → 항상 pinnedIds (망각/backoff 무시)
2) status = active 이고 pinned 아닌 것 중 dueScore ≥ 1 인 후보 수집
3) 정렬키 = dueScore × priorityWeight  (high 3 / mid 2 / low 1) 내림차순
4) 상한 N(기본 9~12개)까지만 rotationIds 채움 — 넘치면 다음 날로
```

> 상한을 둬서 수백 개가 쌓여도 매일 보는 목록은 늘 감당 가능한 크기로 유지된다.

### 5.4 잠듦 전이

```
status active && pinned 아님 && daysSince > dormantThreshold(priority)
  → status = 'dormant', dormantSince = now, prayerPlan에서 제외
```

부활은 "잠든 기도" 화면에서 수동. (자동 삭제는 절대 하지 않음)

    // 2) data.rawText (string), data.knownPeople (string[]) 수신
    // 3) Gemini 호출 → 구조화 JSON 배열 반환
    // 4) 저장하지 않음. 파싱 결과만 반환 (검토는 클라이언트에서)

> **저장은 클라이언트가 검토 후 수행**(결정 #6). 함수는 파싱만 담당해 오분류 위험을 줄인다.

### 6.2 시스템 지시 (요지)

```
당신은 한국어 기도 요청 텍스트를 정리하는 도우미다.
- 통일되지 않은 덩어리 텍스트를 "사람별·항목별"로 분리한다.
- 한 사람이 여러 기도제목을 가질 수 있다 → 항목마다 별도 카드.
- 대상자 이름을 추출한다. 불명확하면 personName="" 로 두고 비워둔다.
- knownPeople 목록과 이름이 같거나 유사하면 그 표기를 그대로 쓴다(중복 매칭).
- category는 self/family/church/ministry/friend/other 중 보수적으로 추정.
- priority는 단서가 명확할 때만 high/low, 평소엔 mid.
- title은 12~20자 내외 핵심 요약, body에는 원문 표현을 보존.
- 추측으로 내용을 지어내지 않는다. 출력은 반드시 JSON 스키마만.
```

### 6.3 출력 스키마

```ts
{
  type: 'object',
  required: ['items'],
  properties: {
    items: {
      type: 'array',
      items: {
        type: 'object',
        required: ['personName', 'category', 'title', 'priority'],
        properties: {
          personName: { type: 'string' },
          category:   { type: 'string', enum: ['self','family','church','ministry','friend','other'] },
          title:      { type: 'string', maxLength: 60 },
          body:       { type: 'string', maxLength: 400 },
          priority:   { type: 'string', enum: ['high','mid','low'] },
          confidence: { type: 'number' },   // 0~1, 낮으면 검토 UI에서 경고 배지
        },
      },
    },
  },
}
```

## 7. 게임화 연동 (결정 #8)

기존 `awardEngine`(onWrite) 패턴에 기도 트리거를 추가한다.

### 7.1 신규 트리거 — `functions/src/prayerAward.ts`

`users/{uid}/days/{date}/prayerChecks/{prayerId}` onCreate 시:

1. `prayers/{prayerId}` 갱신: `lastPrayedAt`, `prayCount += 1`, `streak` 갱신.
2. 포인트 적립 + `pointLedger` 기록.
3. 오늘 `prayerPlan`을 모두 체크했으면 → `prayerStreak += 1`, 완료 보너스.
4. 배지 조건 확인(기도 스트릭 7/30/100 등).

### 7.2 포인트 상수 (제안값 — 시작 시 조정 가능)

```ts
export const PRAYER_POINT_EARN = {
  PRAYER_CHECK:        2,    // 기도제목 1건 체크
  DAILY_LIST_COMPLETE: 15,   // 오늘의 목록 전부 완료
  PRAYER_STREAK_7:     40,
  PRAYER_STREAK_30:    150,
  PRAYER_ANSWERED:     30,   // 응답 기록 시 (간증 작성 권장)
} as const;
```

> 일일 체크 포인트에는 상한(예: 하루 30P)을 둬서 인플레이션 방지.

## 8. 보안 · 인덱스

- **보안 규칙**: 기존 `users/{uid}/{document=**}` 규칙이 모든 하위 컬렉션(people, prayers, prayerChecks)을 이미 커버 → 변경 불필요.
- **복합 인덱스** (`firestore.indexes.json`):
  - `prayers`: `status ==` + `priority` + `receivedAt desc`
  - `prayers`: `personId ==` + `status ==`
  - `prayers`: `status ==` + `category` + `receivedAt desc`
- **Functions Secret**: 기존 `GEMINI_API_KEY` 재사용 → 추가 설정 없음.

## 9. 구현 단계 (제안)

검토 후 코딩에 들어가면 아래 순서로 진행한다.

1. **타입·스키마** — `shared/types/firestore.ts`에 PrayerPersonDoc 확장, PrayerDoc 개편, 상수 추가. 인덱스 정의.
2. **기도 탭 셸** — TabBar에 "기도" 탭, 라우트 `/prayers`, 세그먼트 컨트롤·빈 화면.
3. **빠른 추가 + 전체 목록** — 한 줄 저장, 목록·필터·사람 매칭(가장 자주 쓰는 핵심 먼저).
4. **일일 체크 + 오늘 화면** — prayerChecks, 진행 표시, 체크 인터랙션.
5. **로테이션·망각 + dailyReset 확장** — prayerPlan 계산, 잠듦 전이.
6. **AI 무더기 정리** — `parsePrayerBulk` 함수 + 검토 UI.
7. **응답 보관함 + 잠든 기도** — 상태 전이·간증.
8. **게임화** — `prayerAward` 트리거, 포인트·스트릭·배지.
9. **마이그레이션·시드** — 기존 gratitudes/PrayerDoc 데이터 변환, Admin에 시드 버튼.

각 단계는 독립 배포 가능하도록 구성(1→2→3 순으로 점진 사용 시작).
