# 습관 정원 (Habit Garden) — 프로젝트 계획서

> 본 문서는 기존 `00_프로젝트_계획서.md`(전산화 플래너)를 **대체**하는 새 프로젝트의 공통 컨텍스트다.
> 기존 플래너의 5분 단위 타임테이블 수동 입력이 부담스러워, **습관 교정 중심**으로 완전히 새로 설계한다.
> 핵심 제약: **본인 1인 사용 / 웹 단일 앱(PWA) / 폰 중심 + 멀티기기 동기화 / 빠른 개발(MVP 1.5–2주)**
> 구현은 Claude Cowork으로 진행하며, 디자인은 코딩 전에 Claude로 HTML 목업을 먼저 만들어 다듬는다.

## §0. 문서 사용 안내

### 이 문서가 기존 계획서와 다른 점
- **타임테이블(5분 슬롯) 완전 제거.** 매시간 입력하는 부담을 없앤다.
- **습관 체크 + 하루 1회 회고**를 일일 기록의 핵심으로 둔다.
- **동기부여 레이어 추가**: 포인트·레벨·스트릭, 배지·업적, 가벼운 페널티, 그리고 **식물/정원 성장 메타포**.
- 컨디션·플래너·경건·AI 피드백은 기존 개념을 이어받되, 새 정보구조에 맞게 재배치한다.

### 동반 문서
- `03_시드데이터.md` — 시드 습관 8개, 회고 질문, 포인트 가격표, 식물 종, 배지 정의. 시드 스크립트·`/admin` 기본값의 출처.
- `04_목업_mockups.html` — **확정된 디자인** 목업(메인 대시보드 / 정원). 브라우저로 열어 색·레이아웃·정원 일러스트 기준으로 사용.

### 개발 워크플로우
1. **디자인 우선(완료)**: 메인 대시보드·정원 목업을 제작·확정했다(`04_목업_mockups.html`). 습관 체크·진척 대시보드·회고 시트 목업은 구현 직전 동일 톤으로 추가한다.
2. 목업 확정 후 Cowork으로 단계별 구현. 의존성 없는 작업은 병렬 진행.
3. 각 단계의 "완료 시 동작"을 기준으로 검수 후 다음 단계로 이동.

### 빠른 개발 원칙
- **shadcn/ui 우선**: 검증된 컴포넌트(Dialog, Sheet, Tabs, Button, Slider 등)는 그대로 사용. 정원 시각화 같은 고유 UI에만 커스텀.
- **YAGNI**: 1인용이므로 권한·역할·다국어·접근성 폴리시는 생략.
- **수동 입력 수용**: 컨디션·습관·회고는 수동 입력. 정확도보다 **기록의 지속성**을 우선.
- **타입 안전성**: Firestore 스키마는 단일 `shared/types/firestore.ts`에 정의해 클라이언트·서버 양쪽에서 import.
- **에뮬레이터 적극 사용**: Firebase Emulator Suite로 로컬 빠른 반복.

## §1. 개요

### 무엇을 만드는가
- 손글씨 플래너의 디지털화가 아니라, **습관을 교정하기 위한 개인 전용 PWA**.
- 매일 가볍게 습관을 체크하고 하루를 한 번 돌아보면, 그 성취가 **포인트·레벨·정원의 성장**으로 시각화되어 동기를 유지시킨다.
- AI(LLM)가 매일 데이터를 분석해 **사실 기반 + 동기부여형** 코칭 피드백을 제공한다.

### 설계 철학 (가장 중요)
1. **부담 최소화**: 하루에 필요한 입력은 "습관 체크 + 회고 3~4문항"이 전부. 컨디션은 슬라이더 몇 개. 1~2분 안에 끝나야 한다.
2. **즉각적 보상**: 입력 즉시 포인트 적립·정원 변화·스트릭 갱신이 눈에 보여야 한다.
3. **건강한 동기부여**: 페널티는 게임 메커니즘(스트릭 끊김, 포인트 차감, 성장 일시정지)에 한정한다. **식물이 죽거나 자기비난을 부르는 연출은 쓰지 않는다.** 시든 식물도 다시 회복할 수 있다.
4. **누적의 가시화**: 잔디 히트맵·추세 그래프·정원 서사로 "내가 쌓아온 것"을 한눈에 보여준다.

### 파트 구성
1. **습관 (핵심)**: 1–5점 + 가중치 체크리스트, 시간대별 분류, 달성 기준 기반 스트릭.
2. **회고 (핵심)**: 하루 1회 구조화된 질문 3~4개. 행동 점검 중심.
3. **컨디션**: 수면/에너지/기분 점수, 기상/취침 시각, 날씨(자동 수집).
4. **플래너**: 오늘 할 일, 장기 투두, 캘린더 표시.
5. **경건**: 감사·기도제목·신앙적 성찰. (회고와 역할 분리 — §11 참조)
6. **정원 / 진척 (동기부여 레이어)**: 포인트·레벨·스트릭·배지, 식물/정원 성장, 잔디 히트맵, 추세 대시보드.
7. **AI 피드백**: 어제 데이터 기반 균형형(사실+동기부여) 코칭.

### 사용 환경
- **메인 기기: 갤럭시 S24 Ultra** (확정).
  - 물리 해상도 3120×1440(QHD+), 약 6.8″. 기본 표시는 FHD+(2340×1080)로 동작하는 경우가 많다.
  - 브라우저 논리 폭(CSS px)은 설정에 따라 대략 **384–412px**, DPR 약 3.0–3.5. **본 앱은 380px 폭 기준으로 설계하고 412px까지 우아하게 확장**한다(목업이 380px 기준).
  - Chrome 또는 Samsung Internet에서 "홈 화면에 추가" → `display: standalone` 풀스크린.
  - 펀치홀 카메라/하단 제스처바 대응: `viewport-fit=cover` + `env(safe-area-inset-*)`로 상·하단 안전영역 패딩.
  - SVG/Framer Motion 정원 애니메이션은 고DPR에서도 선명하도록 벡터 자산 사용.
- **보조**: 아이패드, PC 웹. 멀티기기에서 동일 데이터 실시간 동기화.
- **본인 외 접근 차단**: Google Sign-In + 본인 UID 화이트리스트.

## §2. 기술 스택 & 아키텍처

### 추천 근거 (사용자에게 위임받은 결정)
요구사항이 **폰 중심 + 멀티기기 실시간 동기화 + PWA 설치 + 푸시 알림 + 오프라인 입력**이다. 이 조합을 가장 적은 셋업으로 만족하는 것이 **React + Firebase + PWA**다.

- 멀티기기 동기화/오프라인: Firestore가 `onSnapshot` 실시간 구독 + IndexedDB persistence로 기본 제공.
- 푸시 알림(시간대별 적극 알림): Firebase Cloud Messaging + Cloud Functions 스케줄러.
- PWA 설치/풀스크린: vite-plugin-pwa.
- 비용: 1인 사용은 무료 티어 내(월 $0~3 수준, AI 호출 포함).
- 검증성: 기존 플래너에서 이미 검토된 스택이라 리스크가 낮다.

> **대안 검토**: "폰 한 대만" 사용이라면 로컬 우선(IndexedDB only) 구성으로 백엔드를 없앨 수 있었으나, 멀티기기 동기화를 원하셨으므로 클라우드 백엔드가 필요하다. Supabase도 후보지만 Firebase가 푸시·스케줄·호스팅을 한 콘솔에서 묶어주어 1인 프로젝트에 유리하다.

### 클라이언트
- **Vite 5 + React 18 + TypeScript 5.x**
- **Tailwind CSS 3.4 + shadcn/ui** (Radix UI 기반)
- **상태 관리**: Zustand(전역 최소) + TanStack React Query 5(서버 상태)
- **데이터**: Firestore SDK (`onSnapshot` 실시간 구독, 오프라인 캐시 활성)
- **라우팅**: React Router 6
- **시간 처리**: date-fns + date-fns-tz (KST 고정)
- **아이콘**: lucide-react
- **차트**: Recharts (추세 그래프), 잔디 히트맵·정원은 커스텀 SVG
- **애니메이션**: Framer Motion (정원 성장, 포인트 적립 연출)
- **PWA**: vite-plugin-pwa (workbox)

### 백엔드 (서버리스)
- **Firebase**: Firestore(메인 DB) / Auth(Google Sign-In) / Cloud Functions(Node 20, TS) / Hosting / Storage(백업) / Cloud Messaging(푸시)
- **AI**: LLM API(예: Gemini 2.5 Flash 또는 Claude Haiku) — Cloud Functions 내부에서만 호출, 키 클라이언트 노출 금지.
- **외부 API**: 기상청 단기예보 (Cloud Functions에서 호출 후 캐싱) — 컨디션 파트(2단계)에서 도입.

### 배포 & 리전
- 개발: Firebase Emulator Suite.
- 프로덕션: Firebase Hosting + Functions, 리전 `asia-northeast3`(서울).
- CI: GitHub Actions로 main push 시 빌드+배포(선택, 후반).

## §3. 시스템 정책

### 하루의 경계
- **모든 데이터는 KST 04:00을 하루의 시작으로 본다.** (기존 정책 유지)
- 03:59까지의 기록은 "어제"에 귀속.
- 04:00 정각에 DayDoc 생성·어제 마감·스트릭 정산·AI 피드백 생성.
- 클라이언트는 진입 시점에 시계를 비교해 날짜를 보정한다(백그라운드 SW 보장 안 됨 대비).

### 표준 함수 (TypeScript)
```typescript
// src/lib/dayBoundary.ts
import { toZonedTime, format } from 'date-fns-tz';
import { subHours } from 'date-fns';

const KST = 'Asia/Seoul';

/** 주어진 시각이 속하는 '플래너 날짜'를 YYYY-MM-DD로 반환 (04:00 경계) */
export function plannerDate(date: Date = new Date()): string {
  const kst = toZonedTime(date, KST);
  const shifted = subHours(kst, 4);
  return format(shifted, 'yyyy-MM-dd', { timeZone: KST });
}

/** 지금이 속한 시간대(아침/점심/저녁/밤) 반환 — 습관 시간대 분류용 */
export function timeOfDay(date: Date = new Date()): 'morning'|'afternoon'|'evening'|'night' {
  const h = toZonedTime(date, KST).getHours();
  if (h >= 4 && h < 11) return 'morning';
  if (h >= 11 && h < 17) return 'afternoon';
  if (h >= 17 && h < 22) return 'evening';
  return 'night';
}
```

### 동기화 / 오프라인 / 인증
- **Last-Write-Wins** (Firestore 기본). 1인이라 충돌 거의 없음. 모든 문서에 `updatedAt: serverTimestamp()`.
- Firestore IndexedDB persistence 활성 → 오프라인 입력 후 복귀 시 자동 동기화.
- Google Sign-In만 허용. 본인 UID를 Firestore Rules와 Functions에 **하드코딩**. 다른 UID는 전부 거부.

## §4. Firestore 스키마

> 모든 인터페이스는 `shared/types/firestore.ts`에 단일 정의하여 클라이언트·Functions 양쪽에서 import.

```typescript
// shared/types/firestore.ts
import { Timestamp } from 'firebase/firestore';

// ── 일일 문서 ───────────────────────────────────────────
// users/{uid}/days/{YYYY-MM-DD}
export interface DayDoc {
  date: string;                  // 'YYYY-MM-DD' (04:00 경계 기준)
  condition: ConditionData;
  reflection?: ReflectionData;   // 하루 1회 회고 (§10)
  dayScore?: number;             // 습관 가중평균 등으로 산출된 그날 종합 점수 (0-100)
  pointsEarned?: number;         // 그날 적립 포인트 합 (정산 결과)
  streakSnapshot?: number;       // 그날 시점의 전체 스트릭 일수
  aiFeedback?: AIFeedback;
  finalized?: boolean;           // dailyReset가 어제를 마감하면 true
  updatedAt: Timestamp;
}

// ── 컨디션 (기존 유지) ──────────────────────────────────
export interface ConditionData {
  sleepScore?: number;           // 0-100
  energyScore?: number;          // 0-100
  moodScore?: number;            // 1-10
  bedTime?: string;              // 'HH:mm'
  wakeTime?: string;             // 'HH:mm'
  immediatelyAwoke?: boolean;
  sleepEfficiency?: number;      // 0-1
  weather?: WeatherSnapshot;     // fetchWeather가 채움
}
export interface WeatherSnapshot {
  tempMin: number; tempMax: number; rainProb: number;
  feelsLike?: number;
  pm10?: 'good' | 'normal' | 'bad' | 'verybad';
  fetchedAt: Timestamp;
}

// ── 습관 정의 ───────────────────────────────────────────
// users/{uid}/habits/{id}
export interface HabitDoc {
  id: string;
  title: string;
  weight: number;                // 1-10 (가중치)
  timeOfDay: 'morning' | 'afternoon' | 'evening' | 'night' | 'anytime';
  order: number;
  scoreMode: 'scaled' | 'binary';// 'scaled'=1-5점, 'binary'=완료/미완료
  achieveThreshold: number;      // 스트릭 '달성' 판정 기준 (scaled=기본 3, binary=1)
  iconName: string;              // lucide 아이콘
  description?: string;
  active: boolean;
}

// ── 습관 체크 (일일) ────────────────────────────────────
// users/{uid}/days/{date}/habitChecks/{habitId}
export interface HabitCheckDoc {
  habitId: string;
  score: number | null;          // scaled:1-5, binary:0|1, null=pass(건너뜀)
  achieved: boolean;             // score >= achieveThreshold 면 true (스트릭/잔디 판정)
  note?: string;
  checkedAt: Timestamp;
}

// ── 습관별 스트릭 상태 ──────────────────────────────────
// users/{uid}/habits/{id}/streak (단일 문서) 또는 HabitDoc에 임베드
export interface HabitStreakData {
  current: number;               // 현재 연속 달성 일수
  best: number;                  // 역대 최장
  lastAchievedDate?: string;     // 마지막 달성일
  freezeTokens: number;          // 스트릭 보호권 보유 수 (§9 페널티 완화)
}

// ── 회고 (하루 1회 구조화) ──────────────────────────────
export interface ReflectionData {
  answers: Record<string, string>; // questionId → 답변 (§10 질문 세트)
  completedAt: Timestamp;
}

// ── 경건 (신앙 성찰, 회고와 분리) ───────────────────────
// users/{uid}/days/{date}/gratitudes/{id}
export interface JournalEntryDoc {
  id: string;
  text: string;
  createdAt: Timestamp;
}
// users/{uid}/prayers/{id}
export interface PrayerDoc {
  id: string;
  category: 'self' | 'family' | 'church' | 'other';
  title: string;
  body?: string;
  rotationDays: number;          // 0=매일 고정, n=n일 주기
  priority: 'high' | 'mid' | 'low';
  active: boolean;
}

// ── 플래너 ──────────────────────────────────────────────
// users/{uid}/longTodos/{id}
export interface LongTodoDoc {
  id: string; title: string;
  deadline?: string; priority: 'high'|'mid'|'low';
  progress: number; done: boolean;
  createdAt: Timestamp; updatedAt: Timestamp;
}
// users/{uid}/days/{date}/todayTodos/{id}
export interface TodayTodoDoc {
  id: string; title: string; done: boolean;
  linkedLongTodoId?: string;
}

// ── 게임화 / 진척 (동기부여 레이어) ─────────────────────
// users/{uid}/progress (단일 문서)
export interface ProgressDoc {
  totalPoints: number;           // 누적 적립 포인트(평생)
  spendablePoints: number;       // 정원에 쓸 수 있는 현재 잔액
  level: number;
  xpInLevel: number;             // 현재 레벨 내 경험치
  globalStreak: number;          // 전체(하루 단위) 스트릭
  globalBestStreak: number;
```

## §6. 습관 (핵심)

### 상세 페이지 (`/habits`)
- 시간대별 섹션(아침/점심/저녁/밤/언제든)으로 그룹.
- 각 습관 카드: 제목, **1–5점 동그라미 탭** 또는 **완료 토글**(scoreMode에 따라), 가중치 뱃지, pass(건너뜀) 버튼, 메모 펼치기, 현재 스트릭(🔥n).
- 점수 입력 즉시 저장 → `achieved` 판정 → 포인트 적립 토스트 + 정원 변화 반영.
- 하단 요약: **오늘 가중평균 점수(dayScore)** 와 **달성률(달성 습관 수 / 전체)**.

### 평가 방식 — 1–5점 + 가중치
- 대부분 습관은 `scoreMode: 'scaled'` (1–5점). 일부는 `'binary'`(완료/미완료)로 둘 수 있다(예: "운동했다/안했다").
- **가중평균 점수(dayScore)** = Σ(score 정규화 × weight) / Σ(weight) × 100. (pass는 분모에서 제외)
- **스트릭 '달성' 판정 (중요)**: 각 습관은 `achieveThreshold`를 가진다.
  - scaled: 기본 **3점 이상이면 달성**(습관별 조정 가능).
  - binary: 1(완료)이면 달성.
  - `HabitCheckDoc.achieved`에 저장 → 잔디·스트릭·포인트의 단일 기준이 된다.
- **전체(하루) 스트릭**: "그날 가중치 상위 핵심 습관들의 달성률이 기준치(기본 60%) 이상"이면 그날을 '성공한 날'로 보고 globalStreak +1. (개별 습관 스트릭과 별개)

### 시드 습관 (기존 계획서 셋 재사용, /admin에서 수정)
| 제목 | 시간대 | 가중치 | scoreMode | 비고 |
|------|--------|--------|-----------|------|
| QT·아침기도 | 아침 | 10 | scaled | 경건과 별개의 행동 습관 |
| 수업복습 | 저녁 | 8 | scaled | |
| 스마트폰 절제 | 언제든 | 9 | scaled | 디지털 웰빙 확인 후 자가평가 |
| 숏츠 절제 | 언제든 | 9 | scaled | YT/IG/TT 합산 |
| 운동 | 아침 | 7 | binary | |
| 플래너 기록 | 언제든 | 7 | binary | 본 앱 사용 자체 |
| 설거지 | 저녁 | 5 | binary | |
| 청소 | 저녁 | 5 | binary | |

### 자동 측정 불가 항목(자가 평가 가이드)
- 스마트폰 절제: <2h=5, 2–3h=4, 3–4h=3, 4–5h=2, 5h+=1.
- 숏츠 절제: <30m=5, 30–60=4, 60–90=3, 90–120=2, 120+=1.

## §7. 회고 (하루 1회, 구조화)

### 상세 페이지 (`/reflection`)
- 저녁 알림으로 진입 유도. 하루 1회만 작성(수정 가능).
- **구조화된 질문 3~4개**에 짧게 답한다. 기본 질문 세트:
  1. 오늘 가장 잘 지킨 행동 한 가지는?
  2. 가장 아쉬웠던 점과 그 원인은?
  3. 내일 딱 한 가지 더 잘하고 싶은 것은?
  4. (선택) 오늘 컨디션/집중을 한 단어로 표현하면?
- 질문 세트는 `/admin`에서 편집 가능.
- **완료 시 보상**: 회고 작성 자체에 포인트 적립(습관과 별개의 고정 보상) → 매일 돌아보는 행위를 강화.

### 회고 vs 경건의 역할 분리 (의도적 설계)
회고와 경건이 겹치지 않도록 명확히 나눈다.
- **회고(`/reflection`)** = *행동 점검*. 습관·생산성·자기관리 관점의 짧은 자기 검토.
- **경건(`/devotion`)** = *신앙적 성찰*. 감사·기도제목·하나님 앞에서의 잘한/잘못한 일.
- 두 입력이 비슷해 보일 수 있으나 프레이밍과 목적이 다르며, AI 피드백도 둘을 구분해 활용한다.

## §8. 컨디션 · 플래너 · 경건

### 컨디션 (`/condition`) — 기존 그대로
- 점수 슬라이더 3개(수면 0–100, 에너지 0–100, 기분 1–10), 기상/취침 시각, 바로 기상 토글, 수면 효율.
- 날씨 카드(읽기 전용): 기온/강수확률/체감/미세먼지. `fetchWeather` Cloud Function이 채움.
- 입력 즉시 저장(디바운스 500ms). 메인 한 줄에 요약 표시, 미입력 시 `!` 강조.
- **날씨 수집**: 매일 04:00 + 12:00 KST, 기상청 단기예보 API(대구 nx=89, ny=90). 키는 Functions Secret.

### 플래너 (`/planner`)
- **오늘 할 일**: 체크리스트, 빠른 추가, 장기 투두에서 끌어오기.
- **장기 투두**: 카드 리스트(우선순위·진행률).
- **캘린더**: Google Calendar iframe(읽기 전용)으로 MVP 후반 빠르게.
- 할 일 완료도 소액 포인트 적립 대상(선택).

### 경건 (`/devotion`)
- **오늘 기도제목**: 매일 04:00 선정 — `rotationDays=0` 고정분 전부 + high 우선순위 1개 + 카테고리별 로테이션. 메인 한 줄엔 상위 2개.
- **감사 / 잘한 일 / 잘못한 일**: 자유 입력(신앙적 성찰 관점).
- 기도제목 CRUD는 `/admin`.

## §9. 게임화 & 정원 (동기부여 레이어)

> 본 앱의 차별점. 성취를 **포인트·레벨·스트릭·배지**로 환산하고, **식물/정원 성장**으로 시각화한다.
> 포인트는 **정원 안에서만** 소비된다(현실 보상 교환 없음 — 자기완결적 루프).

### 포인트 경제 (적립 ↔ 소비 균형)
모든 적립/소비는 `pointLedger`에 기록해 추적·디버깅 가능하게 한다.

**적립**
| 행동 | 포인트 | 비고 |
|------|--------|------|
| 습관 1개 달성(achieved) | 가중치 × 2 | 가중치 높은 습관일수록 큰 보상 |
| 습관 만점(5점) | +5 보너스 | scaled 한정 |
| 하루 회고 작성 | +20 (고정) | 돌아보는 행위 강화 |
| 그날 '성공한 날' 달성 | +30 (스트릭 보너스) | 전체 스트릭 +1 조건 충족 시 |
| 스트릭 마일스톤(7/30/100일) | +50/+200/+500 | 배지와 동시 |

**소비** (정원에서만)
| 항목 | 비용(예시) | 효과 |
|------|-----------|------|
| 씨앗 심기 | 50 | 새 식물 시작 |
| 물주기/비료(성장 가속) | 20 | 식물 stage 진행 가속 |
| 새 식물 종 해금 | 200~500 | 도감 확장 |
| 장식(돌·연못·울타리) | 100~300 | 정원 꾸미기 |
| 정원 생기 회복 | 80 | 시든 정원 회복(페널티 후) |

> **경제 밸런스 원칙**: 하루 평균 적립(대략 100~180P)으로 2~4일에 식물 하나를 키울 수 있게 초기 가격을 잡고, /admin에서 가격 테이블을 조정한다. 인플레이션 방지를 위해 고가 해금/장식으로 잉여 포인트를 흡수한다.

### 레벨 & 경험치
- XP = 적립 포인트와 동일 소스(또는 별도 계수). 레벨 구간은 점증(예: Lv n 필요 XP = 100 × n^1.5).
- 레벨업 시 새 식물 종 또는 장식 슬롯 해금 → 장기 동기.

### 스트릭
- **개별 습관 스트릭**: `achieved` 연속 일수.
- **전체(하루) 스트릭**: '성공한 날' 연속(§6 기준).
- **스트릭 보호권(freezeToken)**: 마일스톤 보상 등으로 획득. 하루 놓쳐도 1개 소모해 스트릭 유지(주 1회 한도). 완벽주의 부담을 줄이는 안전장치.

### 식물/정원 메타포 (디자인 핵심)
- 정원은 보유 식물들이 자라는 한 화면. 각 식물은 stage(씨앗→새싹→성장→개화→만개)를 거친다.
- **성장 동력**: 습관 달성·회고·스트릭으로 얻은 포인트를 물주기/비료로 투입하면 stage 진행.
- **생기(health) 0–100**: 정원 전반의 활력. '성공한 날'엔 상승, 페널티 시 하락.
- **계절/도감**: 종을 모으는 수집 요소(장기 리텐션).

### 페널티 — 건강하게 (매우 중요)
자기비난을 부르지 않는 게임 메커니즘으로만 구성한다.
- **스트릭 끊김**: 보호권이 없고 '성공한 날'을 놓치면 globalStreak 0으로. (단, best는 보존)
- **포인트 자연 감소(decay) 없음을 기본으로**, 대신 정원 생기(health)가 미달성 누적 시 서서히 하락.
- **시듦(wither)**: 며칠 방치 시 식물이 `witheredSince`로 시든 상태가 되지만 **죽지 않는다.** 생기 회복(80P) 또는 며칠 달성으로 **반드시 되살아난다.**
- 금지: 식물 사망, 진척 영구 삭제, "실패했어요" 류의 비난성 문구, 붉은 경고·해골 등 위협적 연출.
- 메시지 톤: "정원이 물을 기다리고 있어요" 처럼 회복 가능성을 항상 함께 제시.

## §10. 진척 대시보드 (`/progress`) — 시각화

요청하신 네 가지를 모두 담는다.

### 1) 잔디 / 캘린더 히트맵
- GitHub 잔디 스타일. 칸 = 하루, 색 농도 = 그날 dayScore(또는 달성률).
- 토글: 전체 / 특정 습관별 히트맵. 최근 3개월 + 연간 뷰.

### 2) 추세 그래프 (Recharts)
- 습관 가중평균, 컨디션(수면/에너지/기분), 달성률의 7일·30일 이동평균 선그래프.
- 카테고리/습관 토글, 기간 선택.

### 3) 종합 대시보드
- 상단 카드: 오늘 달성률, 현재 레벨/XP 바, 전체 스트릭(🔥), 포인트 잔액, 정원 생기.
- 주간/월간 요약: 평균 점수, 최고/최저 습관, 신규 배지, 적립 포인트.

### 4) 성장 서사 / 정원 메타포
- 정원 화면을 진척의 '서사'로 활용: "23일째 가꾸는 중, 식물 4그루, 도감 6/20".
- 마일스톤 타임라인(첫 개화, 첫 100P, 첫 30일 스트릭 등).

### 배지 / 업적
- 예시: 7일·30일·100일 스트릭, 특정 습관 50회 달성, 회고 30일 연속, 종 도감 완성, 컨디션 7일 연속 기록 등.
- `/progress`에 배지 그리드(획득/미획득), 획득 시 토스트 + 정원 이벤트.

## §11. 알림 (적극적, 시간대별)

### 정책 — 적극적이되 똑똑하게
사용자 선택: **적극적(시간대별)**. 단, 알림 피로로 무시·차단되지 않도록 가드를 둔다.
- **방해금지 시간대**: 기본 23:00–07:00 발송 안 함(/admin 조정).
- **쿨다운**: 동일 유형 알림은 최소 90분 간격.
- **스누즈/끄기**: 알림에서 바로 1시간 미루기 또는 해당 유형 끄기.
- **이미 완료하면 발송 안 함**: 해당 시간대 습관을 다 체크했으면 리마인더 생략.

### 알림 종류
| 알림 | 시점 | 조건 |
|------|------|------|
| 시간대 습관 리마인더 | 아침/점심/저녁/밤 시작 무렵 | 해당 시간대 미체크 습관 존재 시 |
| 회고 리마인더 | 저녁(기본 21:00) | 그날 회고 미작성 시 |
| 스트릭 위험 경고 | 밤(기본 22:00) | '성공한 날' 미달성 + 보호권 없음 |
| 정원 돌봄 | 저녁 | 시든 식물/낮은 생기 존재 시(부드러운 톤) |
| 마일스톤 축하 | 즉시 | 배지/레벨업/스트릭 달성 시 |

### 구현
- Firebase Cloud Messaging Web Push. 로그인 후 권한 요청, FCM 토큰을 `users/{uid}/_meta.fcmToken`에 저장.
- Cloud Function `habitReminders`: Pub/Sub 스케줄러 30분 간격(07:00–23:00). 위 조건·쿨다운·완료여부를 평가해 발송.
- 클라이언트는 알림 미동작 환경 대비 화면 내 배너로 fallback.

## §13. AI 피드백 (균형형: 사실 + 동기부여)

### 모델 & 호출
- LLM(Gemini 2.5 Flash 또는 Claude Haiku). `responseSchema`로 JSON 강제.
- 매일 04:00 `generateFeedback`가 어제 데이터로 자동 생성. 수동 새로고침 일 5회 상한.
- 실패 시 1회 재시도, 두 번 실패하면 fallback 메시지 저장.

### 시스템 메시지 (균형형 톤)
```
당신은 본인 1인의 일일 데이터를 분석하는 코치다. 한국어로 응답한다.
정량 데이터에 근거한 객관적 진술을 우선한다. 시간/점수/횟수 등은 숫자로 표현한다.
과장된 칭찬·공허한 위로·이모지·느낌표는 쓰지 않는다.
단, 진전과 연속성(스트릭, 추세 개선)은 사실에 근거해 한 줄로 인정하고,
다음 행동을 이어가도록 동기를 부여한다("momentum" 필드).
추천 행동은 구체적이고 측정 가능해야 한다.
출력은 반드시 지정된 JSON 스키마를 따른다.
```

### Response Schema
```json
{
  "type": "object",
  "required": ["oneLineSummary","goodPoints","toFix","recommendations","momentum","conditionAnalysis"],
  "properties": {
    "oneLineSummary": { "type": "string", "maxLength": 80 },
    "goodPoints": { "type": "array", "items": { "type": "string" }, "maxItems": 3 },
    "toFix": { "type": "array", "items": { "type": "string" }, "maxItems": 3 },
    "recommendations": { "type": "array", "items": { "type": "string" }, "maxItems": 3 },
    "momentum": { "type": "string", "maxLength": 80 },
    "conditionAnalysis": { "type": "string", "maxLength": 200 }
  }
}
```

### 입력 데이터(예시 구조)
어제의 습관 달성(점수·가중치), dayScore, 컨디션, 회고 답변 요지, 스트릭 현황, 최근 7일 평균을 요약해 전달. (경건 내용은 사적 영역이므로 기본 미전송 — /admin에서 옵트인)

## §14. Cloud Functions

| 이름 | 트리거 | 역할 |
|------|--------|------|
| `dailyReset` | 스케줄 매일 04:00 KST | DayDoc 생성, 어제 마감, 스트릭 정산, 기도제목 선정, 날씨 fetch, AI 피드백 호출 |
| `awardEngine` | Firestore onWrite(habitChecks/reflection) | 포인트·XP·배지·정원 갱신, pointLedger 기록. **서버에서 단일 정산**(클라 조작 방지) |
| `generateFeedback` | callable + dailyReset 내부 | LLM 호출, AIFeedback 저장. 일 5회 상한 |
| `habitReminders` | 스케줄 30분(07:00–23:00) | 시간대 미체크·회고 미작성·스트릭 위험 평가 후 푸시. 쿨다운·방해금지·완료여부 가드 |
| `fetchWeather` | callable + 스케줄 12:00 | 기상청 API → condition.weather (2단계 도입) |
| `monthlyBackup` | 스케줄 매월 1일 03:30 | Firestore → JSON → Storage, 14개월 rolling |

- **공통 가드**: 모든 함수 진입 시 `uid === 'YOUR_UID'` 확인.
- **리전**: 전부 `asia-northeast3`.
- **Secrets**: `LLM_API_KEY`, `KMA_API_KEY`(기상청).

> **설계 주의**: 포인트/레벨/배지 정산은 반드시 `awardEngine`(서버)에서 한다. 클라이언트가 직접 totalPoints를 쓰지 않도록 Rules로 progress 문서 쓰기를 제한하거나 Functions 경유로 강제한다(경제 무결성).

## §15. 개발 단계 & 일정

> 앞서 제기한 "MVP 범위 과다" 리스크를 반영해, **1단계를 습관+회고+게임화+정원 핵심으로 좁힌다.** 나머지는 단계적으로.

### Phase 1 — 핵심 루프 (약 1주)
셋업 → 디자인 토큰 → 인증/라우팅 → 메인 셸 → **습관 데이터·체크 UI** → **회고** → **포인트/스트릭 정산(awardEngine)** → **정원 기본(씨앗·성장)**.
**완료 시 동작**: 로그인 후 습관을 체크하고 저녁에 회고를 작성하면, 포인트가 적립되고 스트릭이 오르며 정원의 식물이 자란다. (멀티기기 동기화 동작)

### Phase 2 — 시각화 & 컨디션 (약 0.5주)
잔디 히트맵 → 추세 그래프 → 종합 대시보드 → 배지 시스템 → 컨디션 입력·한 줄 → 적극적 알림(habitReminders).
**완료 시 동작**: `/progress`에서 누적 데이터를 보고, 시간대별 리마인더와 마일스톤 알림이 도착한다.

### Phase 3 — 보조 & 마무리 (약 0.5주)
플래너(할일/투두) → 경건(기도제목/감사) → AI 피드백 → 날씨 → PWA/오프라인 마감 → 월별 백업 → /admin(습관·기도제목·포인트 가격·질문 편집).
**완료 시 동작**: 매일 04:00 AI 피드백 생성, PWA 설치, 백업 다운로드, 관리 페이지에서 전부 편집 가능.

### 의존성 (요약)
```
셋업 → 토큰 → 인증/라우팅 → 메인 셸 ─┬→ 습관 체크 → awardEngine → 정원
                                      ├→ 회고
                                      └→ 컨디션(P2)
awardEngine → 진척 대시보드/배지(P2) → 알림(P2)
P3: 플래너 ↔ 경건 ↔ AI ↔ PWA ↔ 백업 ↔ admin (대부분 병렬)
```
