## 2. 평소 업데이트 방법 (가장 자주 쓰는 흐름)

코드를 고친 뒤, 로컬 폴더에서:

```bash
# 1) 빌드가 깨지지 않는지 먼저 확인
cd apps/web
npm install            # 처음 한 번만, 또는 의존성 바뀌었을 때
npx vite build         # 성공하면 OK (dist/ 생성)

# 2) 저장소 루트로 가서 커밋 & 푸시
cd ../..
git add -A
git commit -m "변경 내용 설명"
git push origin main
```

푸시 후 **GitHub → Actions 탭**에서 "Deploy to GitHub Pages" 워크플로가 초록불이 되면
1~2분 안에 라이브 사이트에 반영됩니다.

> ⚠️ 빌드 검증은 `npx vite build`를 권장합니다.
> (`npm run build`는 `tsc -b`까지 돌려 타입 에러로 막힐 수 있음. 실제 배포 워크플로는 `vite build`만 사용)

> ⚠️ Functions(백엔드) 코드를 고쳤다면 자동으로 배포되지 **않습니다**. §5 참고.

## 3. 절대 건드리면 안 되는 설정 (이거 바꾸면 사이트 깨짐)

1. **`apps/web/vite.config.ts` 의 `base: '/habit-garden/'`**
   - GitHub Pages는 `/habit-garden/` 하위 경로로 서빙됩니다. base를 지우면 JS/CSS가 404.
   - 저장소 이름을 바꾸면 이 값도 같이 바꿔야 합니다.

2. **`apps/web/src/App.tsx` 는 `HashRouter` 사용**
   - GitHub Pages는 SPA의 deep link 새로고침을 지원하지 않습니다.
   - `BrowserRouter`로 되돌리면 `/habits` 같은 주소 새로고침 시 404.
   - 로그인 후 URL은 `.../#/` 형태가 정상.

3. **Firebase 승인된 도메인**
   - `jonathanblackdoctor.github.io` 가 이미 등록되어 있어야 Google 로그인이 됩니다.
   - 위치: Firebase 콘솔 → Authentication → 설정 → 승인된 도메인.

4. **Firestore 문서 경로는 반드시 짝수 segment**
   - `users/{uid}/progress` (3-segment) 는 **컬렉션** 경로. `doc()` 호출 시 Firestore가 reject.
   - 단일 사용자 데이터는 `users/{uid}/progress/main` 처럼 **마지막에 doc id** 를 붙여 4-segment 로.
   - 이 함정 때문에 한동안 로그인 후 흰 화면 사고가 났습니다 (§7-D 참고). 새 collection 만들 때 반드시 의식하세요.

5. **PWA(`vite-plugin-pwa`)는 `selfDestroying: true` 모드로 켜져 있음**
   - 옛 서비스워커가 모바일에 박혀 새 빌드를 못 받는 사고를 방지하기 위한 "자폭 SW".
   - 매 배포마다 SW가 install되자마자 자기 자신과 모든 캐시를 unregister/삭제합니다.
   - PWA(홈 화면 설치)를 본격 운영하려면 이 옵션을 끄고 정상 PWA 캐시 전략을 다시 설계해야 합니다.
   - 그냥 끄면 사용자 폰에 또 옛 SW가 박힐 위험이 있으니, 다음 배포 한두 번은 `selfDestroying: true` 를 유지하다가 바꾸세요.

## 4. 저장소에 들어있는 워크플로 2개

- **`.github/workflows/pages.yml`** — `main` push 시 자동 실행. `apps/web` 빌드 후 GitHub Pages 배포. (프론트엔드)
- **`.github/workflows/firebase-deploy.yml`** — **수동 실행 전용**. Functions/Firestore 규칙 등 백엔드를 재배포할 때만 Actions 탭에서 "Run workflow" 실행.
  - 실행하려면 저장소 Secrets 에 `FIREBASE_SERVICE_ACCOUNT` (서비스 계정 JSON) 가 설정되어 있어야 합니다.

GitHub Pages 저장소 설정: Settings → Pages → Source = **GitHub Actions** (이미 설정됨).

## 5. Functions(백엔드) 배포 절차

`functions/src/*.ts` 를 수정했다면 프론트 푸시만으로는 배포되지 **않습니다**. 다음을 따로 수행:

1. 변경 사항을 `main`에 push (프론트 워크플로가 먼저 도는 것은 그대로 OK)
2. GitHub → Actions 탭 → **"Firebase Deploy"** 워크플로 선택 → "Run workflow" → main 브랜치 선택 → 실행
3. 초록불이 되면 함수까지 라이브 반영

> 가장 흔한 실수: Functions 로직(예: `awardEngine`, `prayerAward`, `dailyReset`)을 고치고 프론트만 push 후 "왜 점수/스트릭이 안 갱신되지?" 라고 헤매는 케이스. **백엔드는 별도 수동 배포가 필요합니다.**

## 7. 과거 사고 정리 (재발 방지)

### A. JS/CSS 404 — `base` 누락
- 증상: 사이트가 빈 화면, 콘솔에 `.../assets/index-xxx.js` 가 404
- 원인: `vite.config.ts` 의 `base: '/habit-garden/'` 가 빠짐
- 처리: 절대 지우지 말 것 (§3-1)

### B. 새로고침 시 404
- 증상: `/habits` 등 라우트 직접 입력/새로고침 → 404
- 원인: BrowserRouter 사용
- 처리: HashRouter 유지 (§3-2)

### C. 모바일에서 새 배포가 반영 안 됨 (옛 SW)
- 증상: 데스크탑은 새 버전인데 핸드폰만 옛 화면/흰 화면. 새로고침해도 동일.
- 원인: 이전 빌드의 vite-plugin-pwa 서비스워커가 폰에 박혀 새 번들을 차단.
- 처리: `selfDestroying: true` 모드 SW 를 한 번 배포 → 옛 SW 가 새 자폭 SW 로 교체되며 자기 자신 unregister + 캐시 삭제. 사용자 폰에서 한 번만 새로고침하면 끝.

### D. **로그인 후 흰 화면 (Firestore 경로 함정)** — 2026-05-22
- 증상: 로그인은 되는데 메인 화면이 비어 보임 (단색 배경만).
- 원인: `useGarden.ts` 의 `useProgress` 가 `doc(db, 'users', uid, 'progress')` 호출. 3-segment 라 Firestore가 즉시 reject (`"Document references must have an even number of segments"`). onSnapshot 콜백에서 던진 에러가 React 트리 밖이라 ErrorBoundary도 못 잡고, Main이 빈 채로 AppLayout만 남음.
- 백엔드(`awardEngine.ts`, `prayerAward.ts`, `dailyReset.ts`) 도 같은 잘못된 경로를 사용 중이었어서 점수·스트릭 갱신도 동작 안 했음.
- 처리: 클라이언트·백엔드 모두 `users/{uid}/progress/main` 으로 통일.
- **교훈**: 새 collection/문서 추가 시 segment 카운트를 의식. doc 경로는 짝수, collection 경로는 홀수. ESLint 룰이나 helper 함수로 보강 가능.

### E. iOS PWA standalone 에서 로그인 무반응
- 증상: 모바일 Safari standalone(홈 화면 추가)에서 `signInWithPopup` 무반응
- 처리: `signInWithGoogle` 이 popup 실패 시 `signInWithRedirect` 로 자동 폴백 (§6)

## 10. 기도(Prayers) 기능

이전 작업 로그의 "아직 안 올라간 작업" 항목이었지만 **이번 배포에 모두 포함되어 라이브 동작 중**입니다.
관련 파일:

```
apps/web/src/routes/Prayers.tsx
apps/web/src/features/prayers/usePrayers.ts
apps/web/src/features/prayers/PrayerComponents.tsx
apps/web/src/features/prayers/BulkParse.tsx
shared/prayerRotation.ts
shared/types/firestore.ts          # 기도 타입/상수
functions/src/parsePrayer.ts       # 백엔드(아직 수동 배포 필요)
functions/src/prayerAward.ts       # 백엔드(아직 수동 배포 필요)
```

> 기도 점수·스트릭·응답 보상은 백엔드 함수에 의존합니다. Functions 수동 배포(§5) 가 끝나야 완전히 동작합니다.
